# v0.3a Demo

## 正向流程

1. 打开右侧 AI 面板，切到 `Plan` 模式。
2. 输入：`给 nginx 加 gzip 并验证`。
3. 前端调用 `ai_plan_create`，生成 plan card；随后自动调用一次 `ai_plan_step_execute` 跑首个 probe。
4. 继续点击 `执行下一步`，直到遇到 write step。
5. `ConfirmWriteDialog` 弹出，确认对话框展示：
   - `argv[]`
   - `targetFiles`
   - unified diff 预览
   - snapshot 路径和大文件 warning
6. 点击 `确认写入` 后，后端执行：
   - snapshot 已存在
   - SFTP 覆盖写入
   - `verifyTemplate` 对应模板命令（例如 `nginx -t`）
7. verify 成功后，plan 推进到 `action` step（例如 `nginx -s reload`），再次弹出确认对话框展示 `argv[]`。
8. 点击 `确认执行` 后，reload 成功，step 状态进入 `done`，plan 继续推进直至 `done`。

## 反向流程

1. 在 write step 中故意提交一个无效 nginx 配置。
2. 确认写入后，模板 verify 返回非零。
3. 后端自动执行文件级 rollback，并持续发出 `ai:rollback_progress`；不会推进到 action step。
4. UI 上应看到：
   - step 状态进入 `rolled_back`
   - `ai:service_state_warning` 对应的 warning banner
   - snapshot 路径

## 当前实现说明

- v0.3a 当前已覆盖 `probe -> write -> verify -> action` 主链路，以及 `verify 失败 -> rollback` 反向链路。
- `action` step 目前只允许极小白名单状态变更命令，例如 `nginx -s reload` / `systemctl reload nginx`。
- rollback 仅恢复文件内容，不反向恢复服务状态；服务侧只发 warning。
- write/action step 的确认对话框展示的是后端规范化后的 `argv[]`，不是 AI 原始字符串。

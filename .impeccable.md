## Design Context

### Users

Developers and system administrators who manage remote servers via SSH/SFTP. They use TunnelFiles as a daily work tool — browsing remote file systems, transferring files, and running terminal commands. They value efficiency, reliability, and keyboard-driven workflows. They're comfortable with technical interfaces and expect professional-grade tooling.

### Brand Personality

**Precise. Capable. Confident.**

TunnelFiles should feel like a precision instrument — the kind of tool that earns trust through its craftsmanship. Not flashy, not generic. Every pixel should feel intentional. The interface should communicate competence: "this tool knows what it's doing, and so do you."

### Aesthetic Direction

**Visual tone**: Professional tool — clean, structured, confident. Think IDE-grade density with considered spacing. Not a marketing site, not a consumer app.

**Primary reference**: **Ghostty** (Mitchell Hashimoto) — minimalist but not bland. Precise typography, subtle depth through carefully chosen background tints, restrained color that means something when it appears. Clean borders, no unnecessary decoration.

**Secondary references**: VS Code (layout density, panel system), Warp (modern terminal with clear hierarchy), Linear (precision, typography-driven hierarchy).

**Anti-references**: Generic dashboard templates, glassmorphism, gradient text, neon accents, card-heavy layouts where everything looks the same weight. Anything that says "AI made this."

**Theme**: Dark mode primary. The dark theme should be the showcase — rich, layered, not just "gray on darker gray." Light mode available but secondary.

**Key aesthetic goals**:

- **Depth through layers**: Distinct surface levels (background → card → popover) with subtle but perceptible differences
- **Typography-driven hierarchy**: Weight, size, and opacity changes create clear reading order without relying on decorative elements
- **Restrained color**: Color is signal, not decoration. When blue appears, it means something (interactive, active, primary)
- **Intentional spacing**: Varied rhythm — tight groupings for related content, generous separation between sections. No uniform padding everywhere
- **Density without clutter**: Show more information per screen than a consumer app, but with clear visual structure

### Design Principles

1. **Hierarchy through contrast, not decoration** — Use size, weight, and opacity to create visual levels. Don't add borders, shadows, or background changes when typography alone can communicate the difference.

2. **Color is signal** — Restrained palette. Color appears to communicate state (connected, error, active, selected) not to fill empty space. Background tints over borders where possible.

3. **Space has rhythm** — Tight clusters for related elements, generous gaps between groups. Never the same padding everywhere. The spacing itself communicates structure.

4. **Dense but breathable** — Pack information tightly where it serves the user (file lists, connection cards) but leave room around section headers and primary actions. Think IDE density, not dashboard density.

5. **Every detail is earned** — No element exists without purpose. If a border, shadow, icon, or animation doesn't help the user understand or act, remove it.
